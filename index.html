✅
[OWASP 검증 완료] Content Security Policy Cheat Sheet 및 HTTP Headers Cheat Sheet 대조 확인. 모든 디렉티브·권장값이 공식 문서와 일치합니다.

CSP는 브라우저가 스크립트·스타일·이미지·폰트·iframe 등 모든 리소스를 어떤 출처에서 로드할 수 있는지 허용목록으로 정의합니다. 목록에 없는 출처의 리소스는 자동 차단됩니다. XSS로 삽입된 <script>가 실행되려면 허용된 출처여야 하므로, CSP는 XSS의 가장 강력한 2차 방어선입니다. REST API처럼 HTML을 렌더링하지 않는 응답에는 적용이 무의미할 수 있습니다.

⚠️
2025년 기준: 상위 100만 개 사이트 중 약 25%만 의미 있는 CSP를 적용하고 있습니다. unsafe-inline + unsafe-eval이 모두 있으면 CSP가 있어도 XSS 방어 효과가 거의 없습니다. (출처: GF.dev 2026)
🎓
여기까지가 기본기. 한 단계 더 — 시니어가 실제로 신경 쓰는 것들: nonce 기반 CSP는 견고하지만 스크립트마다 nonce를 서버가 일일이 심어줘야 하는 부담이 있습니다. 아래 strict-dynamic·Trusted Types·SRI는 그 다음 단계에서 마주치는 실전 기법입니다 — "Burp 실전 코드" 탭에서 이어서 다룹니다.
CriticalCSP 헤더 자체 없음
Same-Origin Policy만 적용. 인라인 스크립트·외부 스크립트 모두 제한 없이 실행됩니다.
📌 Burp 확인: Response 헤더에서 Content-Security-Policy 검색 → 없으면 취약점 기록
Critical'unsafe-inline' + 'unsafe-eval' 동시 사용
인라인 스크립트 전체 허용 + eval() 허용. 둘 다 있으면 CSP가 있어도 XSS 방어 효과 없음.
📌 값 예: Content-Security-Policy: script-src 'self' 'unsafe-inline' 'unsafe-eval'
High와일드카드(*) 남용 — script-src: *
모든 외부 도메인 스크립트 로드 허용. 공격자가 임의 서버에 악성 스크립트를 올려 삽입할 수 있습니다.
Mediumframe-ancestors 미설정
CSP Level 3에서 frame-ancestors가 X-Frame-Options를 대체합니다. 없으면 클릭재킹 취약점과 연계됩니다.
InfoContent-Security-Policy-Report-Only만 사용
위반을 보고만 하고 차단하지 않습니다. 테스트용으로만 사용. 운영 환경에서는 실제 CSP 헤더 필요.

/* ❌ 취약한 CSP — Burp에서 이렇게 보이면 즉시 기록 */ Content-Security-Policy: script-src 'self' 'unsafe-inline' 'unsafe-eval' * ↑ unsafe-inline: 모든 인라인 스크립트 허용 ↑ unsafe-eval: eval() / Function() 허용 ↑ *: 모든 외부 출처 허용 — 사실상 보호 없음 /* ✅ 권장 CSP (출처: OWASP HTTP Headers Cheat Sheet + CSP Cheat Sheet 2025) */ Content-Security-Policy: default-src 'self'; /* 기본: 자신의 출처만 허용 */ script-src 'self' 'nonce-{랜덤값}'; /* nonce로 인라인 스크립트 허용 */ style-src 'self'; /* 스타일 자신의 출처만 */ img-src 'self' data: https:; /* 이미지는 HTTPS 전체 허용 */ font-src 'self'; /* 폰트 자신의 출처만 */ frame-ancestors 'none'; /* iframe 임베드 완전 차단 */ base-uri 'self'; /* base 태그 인젝션 방지 */ form-action 'self'; /* 폼 전송 대상 자신만 */ object-src 'none'; /* Flash/플러그인 완전 차단 */ upgrade-insecure-requests; /* 페이지 내 http:// 리소스 요청을 https://로 자동 승격 */
⬆️
upgrade-insecure-requests란: HTTPS로 서빙되는 페이지 안에 아직 http://로 하드코딩된 이미지·스크립트 링크가 남아있으면 믹스드 콘텐츠 경고나 리소스 차단이 발생합니다. 이 디렉티브를 켜면 브라우저가 그런 요청을 자동으로 https://로 바꿔서 보냅니다. 이때 브라우저가 요청에 함께 실어 보내는 것이 Burp 인트로 예시에 나온 Upgrade-Insecure-Requests: 1 요청 헤더입니다 — "나는 이 요청을 안전한(HTTPS) 응답으로 승격해도 된다"는 의사표시입니다. 다만 URL 자체를 안전하게 바꿔주는 것이지 HSTS처럼 최초 접속 자체를 강제하진 않으므로, HSTS를 대체하지는 못합니다.
🔑
nonce는 어떻게 만드나: 요청마다 서버가 CSPRNG(암호학적으로 안전한 난수 생성기)로 128비트 이상 랜덤값을 만들어 Base64로 인코딩합니다. 이 값을 응답 헤더의 'nonce-{값}'과, 실행을 허용할 <script nonce="{같은 값}"> 태그에 동일하게 넣어야 브라우저가 해당 스크립트만 예외적으로 실행합니다. 값을 재사용하거나 예측 가능하게 만들면(예: 순차 증가) 공격자가 값을 추측해 무력화할 수 있으므로 반드시 매 응답마다 새로 생성해야 합니다.

심화 1 — strict-dynamic: nonce/hash로 허용한 스크립트가 자기 자신이 동적으로 추가하는 다른 스크립트(예: analytics 로더가 붙이는 2차 스크립트)에게 신뢰를 전파합니다. 이걸 쓰면 script-src에 도메인을 일일이 나열할 필요가 없어져 대규모 서비스에서 CSP 유지보수 부담이 크게 줄어듭니다. 단, 구형 브라우저 호환을 위해 'self' https:를 함께 적어두면 strict-dynamic을 지원하는 브라우저는 이를 무시하고 strict-dynamic만 적용합니다.
/* ✅ strict-dynamic — 대규모 서비스의 실전 CSP 패턴 (출처: CSP Level 3 스펙, web.dev) */ Content-Security-Policy: script-src 'nonce-{랜덤값}' 'strict-dynamic' 'self' https:; /* 'self' https: → strict-dynamic 미지원 구형 브라우저용 폴백, 지원 브라우저는 무시 */

심화 2 — Trusted Types (DOM XSS 최종 방어선): CSP의 script-src는 "어디서 온 스크립트인가"만 통제합니다. 하지만 브라우저가 문자열을 곧바로 마크업으로 해석해버리는 위험한 DOM API에, 신뢰할 수 없는 문자열을 검증 없이 그대로 흘려 넣는 DOM 기반 XSS는 script-src로 막을 수 없습니다. Trusted Types는 그런 API들에 반드시 "사전에 등록한 정책 함수를 통과한 값"만 넣도록 브라우저 레벨에서 강제해서, 코드 리뷰로도 놓치기 쉬운 DOM XSS 패턴을 원천 차단합니다. 2025년 기준 Chrome/Edge 지원, Firefox/Safari는 아직 실험 단계라 report-only로 먼저 도입하는 것이 실전에서 안전합니다.
/* ✅ Trusted Types 강제 (출처: web.dev, W3C Trusted Types 스펙) */ Content-Security-Policy: require-trusted-types-for 'script'; trusted-types my-app-policy; /* 등록된 my-app-policy를 거치지 않은 값을 위험 Sink에 대입하면 브라우저가 즉시 예외를 던져 차단 */ /* 정책 등록은 페이지 JS에서: trustedTypes.createPolicy('my-app-policy', { createHTML: sanitizeFn }) */
🎓
심화 3 — Subresource Integrity(SRI), CSP의 단짝: CSP가 "이 CDN 도메인은 신뢰한다"고 허용해도, 그 CDN 자체가 해킹되거나 MITM으로 파일이 변조되면 무용지물입니다. SRI는 스크립트/스타일 파일의 해시를 미리 박아둬서, 실제 받아온 파일의 해시가 다르면 브라우저가 아예 실행을 거부하게 만듭니다 — CSP(출처 통제) + SRI(내용 무결성 통제)를 함께 쓰는 것이 서드파티 스크립트를 안전하게 쓰는 정석입니다. SRI는 HTTP 헤더가 아니라 태그 속성이라는 점에 주의하세요.

브라우저에게 max-age 기간 동안 해당 도메인은 무조건 HTTPS로만 접속하라고 지시합니다. HTTP 요청 시 브라우저가 자동으로 HTTPS로 전환합니다. RFC 6797 표준. HTTPS 응답에서만 유효합니다.

방어 공격

max-age=63072000; includeSubDomains; preload (2년)
🚨
왜 preload가 필요한가 — "최초 접속" 취약점: HSTS는 브라우저가 한 번이라도 HTTPS로 접속해 이 헤더를 받아본 뒤부터만 작동합니다(Trust On First Use). 즉 사용자가 그 도메인에 처음 접속하는 순간 HTTP로 요청하면, 이 최초 요청은 HSTS 보호 없이 그대로 전송되어 SSL Stripping에 노출됩니다. preload는 이 최초 접속 취약점을 없애기 위해 도메인을 Chrome·Firefox 등 브라우저에 하드코딩된 목록에 등록하는 것으로, 등록되면 사용자가 한 번도 방문한 적 없어도 첫 요청부터 HTTPS로 강제됩니다.
⚠️
TLS 인증서가 만료되거나 취소된 상태에서 HSTS가 설정되어 있으면 사용자가 사이트에 접근 불가합니다. preload 목록은 hstspreload.org에서 신청하며, 한번 등록되면 브라우저 배포판에 박혀서 제거에 수개월이 걸릴 수 있으므로 신청 전 includeSubDomains까지 포함해 모든 서브도메인의 HTTPS 지원을 반드시 확인해야 합니다.

여기부터는 보안 취약점과 직접 관련 없는, HTTP 통신에서 흔히 쓰이는 일반 헤더입니다. Burp에서 트래픽을 볼 때 자주 마주치므로 참고용으로 정리했습니다. (출처: MDN Web Docs, RFC 9110/9111/9112)
요청 기본 · 식별
헤더방향설명예시
Host요청요청 대상 호스트명+포트. HTTP/1.1부터 필수이며 가상 호스팅(한 서버에 여러 도메인)의 기준이 됨Host: example.com
User-Agent요청클라이언트 소프트웨어 식별 (브라우저·OS·봇 등)User-Agent: Mozilla/5.0 ...
Sec-CH-UA 계열요청User-Agent 문자열을 그대로 파싱해 지문 추적하는 것을 줄이려는 최신 대안(Client Hints). 브라우저/버전/모바일 여부 등을 필요할 때만 별도 헤더로 제공Sec-CH-UA-Platform: "macOS"
Referer요청직전 페이지 URL (스펙 오타로 굳어진 철자). 보안 관점은 Referrer-Policy 참고Referer: https://example.com/page
Origin요청요청을 보낸 출처(scheme+host+port). CORS·POST 요청에서 주로 검증 대상Origin: https://example.com
Date요청/응답메시지가 생성된 시각Date: Mon, 06 Jul 2026 09:00:00 GMT

헤더방향설명예시
Accept요청받을 수 있는 미디어 타입과 우선순위(q값)Accept: text/html,application/xhtml+xml;q=0.9
Accept-Language요청선호 언어와 우선순위Accept-Language: ko-KR,ko;q=0.9,en;q=0.8
Accept-Encoding요청클라이언트가 지원하는 압축 방식Accept-Encoding: gzip, deflate, br
Accept-Charset요청선호 문자셋 — 현재는 사실상 UTF-8로 고정되어 거의 쓰이지 않음Accept-Charset: utf-8
Vary응답같은 URL이라도 여기 나열한 요청 헤더(예: Accept-Encoding)의 값이 다르면 캐시가 별도 응답으로 구분해서 저장해야 함을 지시 — 없으면 압축된 응답이 압축을 지원하지 않는 클라이언트에게 캐시로 잘못 전달될 수 있음Vary: Accept-Encoding, Accept-Language

ETag응답리소스 버전을 나타내는 식별자(보통 해시값) — If-None-Match와 짝ETag: "33a64df551425fcc"

If-Modified-Since요청지정 시각 이후 변경된 경우에만 응답If-Modified-Since: Tue, 15 Jun 2026 09:00:00 GMT

콘텐츠 전송 · 범위 요청
헤더방향설명예시
Content-Length요청/응답본문 크기(바이트)Content-Length: 4829
Content-Encoding응답본문에 적용된 압축 방식Content-Encoding: gzip
Content-Language응답본문의 자연어Content-Language: ko
Transfer-Encoding응답청크 단위 전송 여부 — 전체 크기를 모를 때 사용, Content-Length와 동시 사용 불가Transfer-Encoding: chunked
Range요청리소스의 일부 구간만 요청 (다운로드 재개, 동영상 탐색 등)Range: bytes=200-1000
Accept-Ranges / Content-Range응답서버가 범위 요청을 지원하는지, 실제로 보낸 범위가 전체 중 어디인지Content-Range: bytes 200-1000/5000
🎓
심화 — Content-Encoding과 BREACH 공격: HTTPS로 암호화돼 있어도 gzip 압축은 "같은 문자열이 반복되면 더 잘 압축된다"는 특성이 있습니다. 만약 응답 본문에 공격자가 조작 가능한 값(예: 검색어를 그대로 반사하는 부분)과 CSRF 토큰 같은 비밀값이 같은 압축 문맥 안에 함께 있으면, 공격자가 여러 후보 문자열을 넣어보며 압축된 응답 크기 변화를 관찰해 비밀값을 한 글자씩 유추할 수 있습니다 — 이게 BREACH 공격입니다(2013년 공개, CRIME 공격의 압축 대상을 요청 헤더에서 응답 본문으로 확장한 버전). 대응은 CSRF 토큰처럼 민감한 값을 압축 대상에서 빼거나, 페이지마다 토큰을 무작위 패딩과 함께 매번 다르게 만들어 압축률 관찰을 무의미하게 만드는 것입니다. Burp에서 압축이 켜진 응답에 반사된 사용자 입력과 토큰이 공존하면 이 공격의 후보로 의심할 만합니다.

헤더방향설명예시
Connection요청/응답이 연결을 유지할지 여부Connection: keep-alive
Keep-Alive응답연결 유지 타임아웃과 최대 요청 수Keep-Alive: timeout=5, max=1000
Upgrade요청/응답다른 프로토콜로 전환 요청 (예: HTTP → WebSocket)Upgrade: websocket
Expect요청큰 본문을 보내기 전 서버가 요청을 받아줄지 먼저 확인 — 서버가 100 Continue로 응답해야 클라이언트가 실제 본문을 전송. 일부 프록시·WAF가 이 흐름을 잘못 처리해 요청이 멈춘 것처럼 보이는 원인이 되기도 함Expect: 100-continue
Location응답리다이렉트 대상 URL(3xx 응답, 예: 301/302) 또는 새로 생성된 리소스 위치(201 Created)Location: /login
Allow응답해당 리소스가 지원하는 HTTP 메서드 목록 — 지원하지 않는 메서드로 요청했을 때의 응답인 405 Method Not Allowed와 함께 옴Allow: GET, POST, HEAD
Retry-After응답언제 다시 요청해도 되는지 — 너무 많은 요청을 보냈을 때의 429 Too Many Requests, 서버 점검 중인 503 Service Unavailable 응답과 함께 옴Retry-After: 120

Forwarded요청프록시를 거쳤는지, 원본 클라이언트가 누구인지 표준화된 형식으로 전달 (RFC 7239)Forwarded: for=203.0.113.1;proto=https
X-Forwarded-For요청비표준이지만 사실상 표준. 클라이언트 원본 IP — 클라이언트가 직접 위조 가능하므로 신뢰 전 검증 필요X-Forwarded-For: 203.0.113.1
